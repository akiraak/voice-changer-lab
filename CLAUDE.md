# CLAUDE.md

このファイルは Claude Code がこのリポジトリで作業するときの方針メモです。

## このプロジェクトについて

配信などで使う**リアルタイム声変換**の手法・ツールを調査・検証するための個人ラボ。
詳細な目的とスコープは [README.md](./README.md) を参照してください。

要点だけ:

- リアルタイム実行が前提 (オフライン専用ツールはスコープ外)
- ローカルGPUで動かす OSS を中心に、必要に応じて商用サービスにも触れる
- OBS・配信ソフト・仮想オーディオデバイスとの接続まで扱う

## やり取りの言語

- ユーザーとのやり取り・README/ドキュメントは**日本語**で書く
- コード内のコメントやコミットメッセージは英語/日本語どちらでも可 (既存に合わせる)

## 作業のときに意識すること

- まだ初期段階のリポジトリ。ディレクトリ構成や慣習は **README の「構成 (予定)」を出発点に、必要になってから作る**。架空の構造を勝手に増やさない。
- ファイルの置き場所:
  - 作業着手時のプランファイル → `docs/plans/`（完了したら `docs/plans/archive/`）。詳細は下の「作業着手ルール」参照
  - ツールごとの調査メモ・仕様 → `docs/specs/`
  - セットアップ手順や実測ログ → `experiments/`
  - タスク管理は `TODO.md` / `DONE.md`（vibeboard で編集する）
- ベンチマーク値や「遅延 ◯◯ ms」のような**実測値はユーザーが計測したもの以外を書かない**。一般論として知られている特性と、自分達で確かめた値は明確に分ける。
- モデル・音声素材を扱うタスクが出てきたら、**配布元のライセンス/利用規約を必ず確認**することを促す。第三者の声を許諾なく再現するような使い方は提案しない。

## 触れがちな関連ツール (参考)

調査対象になりやすいもの。実際に検証したかどうかとは別物なので、未検証のものを「使える」と断定しない:

- リアルタイム声変換 OSS: RVC, w-okada/voice-changer, Beatrice, so-vits-svc (リアルタイム派生)
- 商用/有償: Voidol, Voicemod, CoeFont (声質変換系機能), MetaVoice 等
- 配信補助: OBS Studio, VB-CABLE, VoiceMeeter Banana/Potato

## やらないこと

- 第三者を欺く目的・なりすまし目的での声変換手順の作成
- ライセンス未確認のモデル/音声データの取得を伴うスクリプトの作成

## 開発管理画面 (vibeboard)

ローカル開発時のタスク・プラン管理は [vibeboard](https://github.com/akiraak/vibeboard) で行う。
プロジェクト直下に degit で vendor してある（`./vibeboard/`）。

```bash
# 親プロジェクト直下から
node vibeboard/dist/cli.js --root .
```

`http://localhost:3010` でプロジェクト直下の `docs/plans/`・`docs/specs/`・`TODO.md`・`DONE.md` を閲覧・編集できる。

- `TODO` タブで `TODO.md` / `DONE.md` をプレビュー表示・編集できる
  - 編集は楽観ロック（mtime チェック）付き。外部で先に更新されていた場合は保存時に 409 を返し、リロード / 手元維持 / 強制上書き を選べる
  - `fs.watch` + 2 秒ポーリングで外部変更を検知し、SSE でクライアントへ即時反映する
- ローカル開発専用（本番管理画面とは独立）
- ポート変更は `--port` または `VIBEBOARD_PORT` 環境変数で指定可能

## タスク管理ルール

- タスクは `TODO.md` で管理する
- タスクが完了したら `TODO.md` から該当項目を削除し、`DONE.md` に移動する
- `DONE.md` には完了日を `YYYY-MM-DD` 形式で付けて記録する
- 新しいタスクが発生したら `TODO.md` の適切なセクションに追加する
- タスクの実施前に `TODO.md` を確認し、優先度の高いものから着手する
- コミット時に `TODO.md` を確認し、実装した機能に対応するタスクがあれば `DONE.md` に移動する

## 作業着手ルール

作業（実装・調査いずれも）を始めるときは、コードに手を入れる前に以下を行う。

1. **プランファイルを作成する**: `docs/plans/<task-name>.md` に実装プラン or 調査プランを作成する
   - 目的・背景、対応方針、影響範囲、テスト方針を最低限記載する
   - 複数 Phase / Step に分かれる場合はファイル内でも Phase / Step を明示する
2. **`TODO.md` に該当項目があるか確認する**
   - 無ければ適切なセクションに追加する
   - 既存項目があれば、その項目に作成したプランファイルへのリンクを追記する（例: `[plan](docs/plans/<task-name>.md)`）
3. **複数 Phase / Step がある場合は `TODO.md` に子タスクとして追加する**
   - 親項目の下にインデントしたチェックボックスで Phase / Step を列挙する
   - Phase / Step が完了するごとにチェックを入れ、全完了で親項目を `DONE.md` に移す
4. **作業完了時の後片付け**
   - 親タスクを `DONE.md` に移動する
   - 対応するプランファイルは `docs/plans/archive/` に移動する
