# experiments

リアルタイム声変換ツールの**セットアップ手順・実測ログ**を残すディレクトリ。`docs/specs/` で整理した仕様と対になる「自分達の手元で動かして確かめた記録」を置く。

リポジトリ全体の方針は [../CLAUDE.md](../CLAUDE.md) / [../README.md](../README.md) を参照。

## このディレクトリで扱うもの

- ツール / ホスト別の実測ログ（レイテンシ・GPU 使用量・VRAM・音質メモなど）
- セットアップ手順の実走り記録（インストール・初期設定で詰まった点・回避策）
- 計測テンプレ（実測値を一切含まない骨格。後続の実測タスクの足場）

扱わないもの:

- 仕様・公式ドキュメントの要約 → `docs/specs/<tool-name>.md`
- 作業着手時のプランファイル → `docs/plans/`（完了後は `docs/plans/archive/`）
- モデルファイル本体（`.pth` / `.onnx` / `.safetensors` 等）— [`.gitignore`](../.gitignore) で除外済み

## ディレクトリ構成

ツール / ホスト単位でサブディレクトリを切る。

```
experiments/
├── README.md                       # このファイル
└── <tool-name>/
    ├── README.md                   # そのツールの運用ルール（命名規約・記入ルール）
    ├── _template.md                # 実測ログのテンプレ（値を一切含まない骨格）
    └── YYYY-MM-DD_<model>_<env>.md # 個別の実測ログ
```

現存するもの:

- [w-okada-voice-changer/](./w-okada-voice-changer/) — VCClient 経由でモデルを動かした実測ログ

新しいツールの実測を始めるときは、まず `<tool-name>/README.md` と `<tool-name>/_template.md` を整備してから個別ログを書き始める。テンプレ整備は対応する `docs/plans/` のプランファイルで段取りを切る。

## 共通の記入ルール

[../CLAUDE.md](../CLAUDE.md) の「実測値はユーザーが計測したもの以外を書かない」方針を、このディレクトリでは以下のように具体化する。

- **計測値はユーザー本人が計測したものだけ書く**。一般論として知られている数値や、他者のブログ・配布ページから引用した値はここには書かない（引用が必要な場合は `docs/specs/` 側で「公式値」「他者計測」と明示して扱う）
- **計測方法を併記しない値は書かない**。レイテンシも GPU 使用率も VRAM 使用量も CPU 負荷も、「何のツールでどう測ったか」「計測時の負荷状況」をセットで書く
- **未計測項目は空欄にせず `未計測` と明示**する。テンプレ上で「その条件では存在しない項目」は `n/a` と書き分ける
- **モデル / 音声素材のライセンス・利用条件は計測前に必ず確認**する（[../CLAUDE.md](../CLAUDE.md) 方針）。確認していないモデルでの計測ログはこのディレクトリに残さない
- **観察事実と推論は分けて書く**。推論には「推測」「未確認」を明示する

## ファイル命名規約

個別の実測ログは各ツールサブディレクトリ配下に置き、ファイル名は以下の形式に揃える:

```
YYYY-MM-DD_<model>_<env>.md
```

- `YYYY-MM-DD` — 計測日（ISO 8601 形式）
- `<model>` — 主に計測したモデル名（例: `rvc-v2-someone` / `beatrice-v2-someone`）。VCClient 等のホスト名はここに入れない（ツールサブディレクトリ自体がそれを表すため）
- `<env>` — 計測環境の識別子（例: `winpc-rtx4090` / `mac-m2pro`）

複数モデルを横並びで比較する計測では `<model>` を `compare` 等にして、本文側でモデルリストを明記する。

## テンプレ運用

各ツールサブディレクトリ直下の `_template.md` は**値を一切含まない骨格**として維持する。

- テンプレを使うときは `_template.md` をコピーして上記の命名規約でリネームし、項目を埋めていく
- テンプレ自体の項目追加・廃止は、対応する `docs/specs/` 側の更新とセットで行う。テンプレだけ先行して項目を増やさない
- テンプレが過剰に細かくなって埋まらなくなったら、計測ログを書いた人が項目を見直す（プランファイル経由）
