# 調査プラン: クラウド SaaS のリアルタイム入力対応棚卸し

## 目的・背景

[voice-changer-types](../specs/voice-changer-types.md) §2.3 クラウド SaaS / API は、現時点では「`CoeFont の声質変換系機能、Resemble AI（Live / Realtime 系プロダクト）、ElevenLabs（Voice Changer 機能）、Voice.ai のクラウド経路` などが代表例として挙げられるが、**配信ソフトに流せるリアルタイム入力ストリームとして提供されているか**は各サービスで条件が異なり、TTS / 録音後変換が中心の製品も多い」という**名前出しのみ**の状態で止まっている。同 [§未確定事項](../specs/voice-changer-types.md#未確定事項-phase-3-以降で再評価) でも「§2.3 クラウド SaaS にリアルタイム入力を流し込むユースケースが現実的かは、実例が出てきた時点で再評価する」と明示されている。

このタスクは、§2.3 を**個別検証可能な解像度まで上げる**ことを目的とする。具体的には、

- 各 SaaS の**公式情報源**で「**マイク入力 → 別声へリアルタイム変換 → 配信ソフトに流せるストリーム出力**」というユースケースが公式機能として提供されているかを裏取りする
- 「リアルタイム TTS（テキスト → ストリーミング音声）」「録音アップロード → 変換ダウンロード（バッチ）」「会話/通話 SDK 向けの S2S」「配信用途のローカル仮想オーディオ出力」のどこに該当するかを**機能粒度で切り分ける**
- 配信用途で「そもそも候補に入るか」の足切り判断（[voice-changer-types §2.3](../specs/voice-changer-types.md#23-クラウド-saas--api)）の根拠を、棚卸し表として 1 ファイルに集約する

このドキュメントが揃うと、

- voice-changer-types §2.3 の代表例リストを、**「配信リアルタイム入力に乗せられる / 乗せられない / 個別 SDK 経由なら可能 / 未確認」のラベル付き棚卸し**に更新できる
- 同 §未確定事項の「リアルタイム入力ユースケースが現実的か」項目に対し、**公式情報源ベースの一次判断**を反映できる
- 個別 SaaS の深掘り（プラン / 料金プラン / API 仕様）が必要になった時点で、本書を起点に**個別タスクへ昇格させる判断**ができるようになる
- 商用 SaaS は OSS 系（RVC / so-vits-svc / Beatrice / w-okada）と比べて**ライセンス・第三者声利用・配信での利用範囲**の論点が重いため、[CLAUDE.md](../../CLAUDE.md) 方針との照合根拠を残す

## スコープ

含める:

- 棚卸し対象 SaaS（Phase 1 で確定。少なくとも以下 4 件を起点とする）
  - **CoeFont**（声質変換系機能）
  - **Resemble AI**（Live / Realtime / Streaming 系プロダクト）
  - **ElevenLabs**（Voice Changer / Speech-to-Speech 系機能）
  - **Voice.ai**（クラウド経路。デスクトップクライアントは [voice-changer-types §2.2](../specs/voice-changer-types.md#22-ローカル商用ソフトウェア) 側の話なのでここではクラウド経路に絞る）
- 各 SaaS について、**公式サイト / 公式ドキュメント / 公式 API リファレンス / 公式プライシング**で確認できる範囲での
  - 「**配信リアルタイム入力**」に該当する機能の有無（マイク入力 → 別声出力をストリームで取り出せるか）
  - 機能の**提供形態**（Web アプリ / デスクトップクライアント / REST API / WebSocket・gRPC ストリーミング API / 通話 SDK の埋め込み）
  - 配信ソフト（OBS 等）へ**仮想オーディオデバイス経由で流せる経路があるか**（ベンダー側のデスクトップアプリが出力を仮想ケーブルに渡せるか、または開発者がそれを組む API がブラウザ/SDK 越しに提供されているか）
  - 公称レイテンシ（**ベンダーが公式に記載している値のみ**。あくまで「ベンダー公称」として扱い、本ラボでの実測値とは混ぜない / 一覧表ではベンダー名と値を明記）
  - モデル形態（[voice-changer-types §3](../specs/voice-changer-types.md#3-モデル形態の軸) のどれに該当するか。プリセット固定 / Zero-shot / カスタム学習）
  - 料金体系の大枠（無料枠 / 月額 / 従量 / エンタープライズ別建て、配信用途で実用になる規模感の料金がどのプランか）
  - **利用規約上の制約**（商用配信での利用可否、配信収益化可否、第三者の声を再現するクローニングの可否、クレジット要件）
- voice-changer-types §2.3 への反映方針（代表例リストをラベル付き棚卸しに置き換えるか、本書からのリンクで参照する形にするか）
- 後続タスク起票（個別 SaaS の深掘りが必要になりそうな項目、料金プランの実測比較、配信での実利用検証の前提条件など）

含めない:

- **実測のレイテンシ / 音質 / 通信品質**（[CLAUDE.md](../../CLAUDE.md) 方針。ベンダー公称値の引用はベンダー名・出典付きで明記するが、本ラボでの実測値とは別物として扱う）
- 各 SaaS の**TTS 機能の完全カタログ化**（TTS は本書のスコープ外。「リアルタイム声変換 / S2S」に直接関わる範囲のみ扱う）
- 各 SaaS の**録音後バッチ変換ワークフロー**（配信のリアルタイム入力に乗せられるかが本書の主題なので、バッチは「リアルタイム不可」のラベルだけ付けて深掘りしない）
- **第三者の声を許諾なく再現する手順**および**特定キャラクター音声へのクローニング誘導**（[CLAUDE.md](../../CLAUDE.md) 方針。「クローニング機能の有無 / 利用規約上の制約」は書くが、具体的な再現手順は書かない）
- **SaaS ベンダーが提供する SDK のコード書き方ガイド**（必要になった時点で個別タスクに切り出す）
- 棚卸し対象に**新規 SaaS を片っ端から追加すること**（Phase 1 で対象集合を確定し、本書は確定した集合に集中する。スコープ外の有力候補が出てきた場合は §未確認事項 / 後続タスクに名前だけ残す）

## 対応方針

成果物は `docs/specs/cloud-saas-realtime.md` の 1 ファイル。
冒頭で本書が voice-changer-types のどこを掘り下げるか（§2.3 クラウド SaaS / API の代表例棚卸し）を 1 行で示す。個別ツール spec（[rvc.md](../specs/rvc.md) / [so-vits-svc.md](../specs/so-vits-svc.md) / [beatrice.md](../specs/beatrice.md) / [w-okada-voice-changer.md](../specs/w-okada-voice-changer.md)）とは異なり、**1 ツール深掘りではなく複数 SaaS の横断棚卸し**である点を冒頭で明示する。

構成案:

1. 概要（本書の位置付け / 対象 SaaS / 「リアルタイム入力対応」の定義 — 何を満たしたら「配信に乗せられる」と判定するかの基準を最初に固定する）
2. 評価項目一覧（後述の棚卸し表のカラム定義。`リアルタイム VC 機能の有無 / 提供形態 / 配信ソフト経路 / ベンダー公称レイテンシ / モデル形態 / 料金体系 / 利用規約上の制約` を縦軸の SaaS × 横軸の項目で表す）
3. SaaS 別エントリ（CoeFont / Resemble AI / ElevenLabs / Voice.ai）。各エントリは
   - 公式情報源へのリンク（公式サイト / 公式ドキュメント / プライシングページ）
   - 「リアルタイム声変換」に該当する公式プロダクト名 / 機能名（**ベンダーが何と呼んでいるか**をそのまま書く。本ラボの分類に勝手に翻訳しない）
   - 機能粒度の切り分け（リアルタイム VC / リアルタイム TTS / バッチ VC / クローニングのみ など）
   - 配信用途で乗せる場合の経路（デスクトップクライアント / 開発者 API / WebSocket / 通話 SDK 埋め込み のどれが該当するか）
   - 料金体系の大枠と、配信での実用ラインがどのプランかの推定（プラン名・公称価格を出典付きで）
   - 利用規約上の制約（**配信での利用可否 / 商用配信での利用可否 / 第三者声のクローニング可否 / クレジット要件**を 1 ブロックにまとめる）
   - 未確認事項（公式ドキュメントから読み取れなかった項目）
4. 横断棚卸し表（§2 で定義した評価項目のマトリクス。配信用途で**そもそも候補に入るか / 入らないか**の足切り判断が一目で分かる形）
5. voice-changer-types §2.3 への反映方針（代表例リストを本書の棚卸しでどう更新するか。具体的な更新差分の案までを書く）
6. 配信用途で見えてきた論点（OSS との比較で SaaS が向く / 向かないユースケース、ネットワーク前提・料金前提が配信のリアルタイム要件と噛み合うかの一次判断）
7. 未確認事項 / 後続タスク（個別 SaaS の深掘りが必要そうなもの、料金プランの実測比較、配信での実利用検証、棚卸し対象の追加候補）

記述ルール（[rvc.md](../specs/rvc.md) / [so-vits-svc.md](../specs/so-vits-svc.md) / [beatrice.md](../specs/beatrice.md) / [w-okada-voice-changer.md](../specs/w-okada-voice-changer.md) と統一）:

- **公式サイト / 公式ドキュメント / 公式 API リファレンス**に書かれていることと、コミュニティで一般に言われていることを明確に分ける。出典を行ごとに明記する
- 実測値・断定的な性能評価は書かない（[CLAUDE.md](../../CLAUDE.md) 方針）。**ベンダー公称レイテンシは「ベンダー公称」として明示**し、本ラボでの実測値とは混ぜない
- 機能名・プロダクト名は**ベンダーがそう呼んでいる名称**をそのまま書く（本ラボの分類軸へは別カラムで対応付ける）
- 第三者キャラクター音声 / 特定声優のクローニング手順は書かない（言及はしても、具体的な再現導線・リンクは記載しない）
- 「リアルタイム声変換」と「リアルタイム TTS」を**同じカラムに混ぜない**。本書のスコープはリアルタイム声変換側のみで、TTS は「スコープ外」として明示扱い
- SaaS ごとに**最終確認日**を明記する（プライシング・規約は変動が早いため）
- 料金は USD / JPY / 月額 / 従量 を**ベンダー表記そのまま**で書き、為替換算は行わない
