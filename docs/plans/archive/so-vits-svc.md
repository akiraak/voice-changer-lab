# 調査プラン: so-vits-svc の仕様

## 目的・背景

[voice-changer-types](../specs/voice-changer-types.md) §1.3 ニューラル変換（**コンテンツ表現 + 話者条件付け系**）/ §2.1 ローカル OSS / §3.1 話者依存型 に位置付けた **so-vits-svc** を、個別ツールとして掘り下げる。

so-vits-svc は本ラボでの位置付けとして、

- [RVC](../specs/rvc.md) と並んで OSS 系で実運用例が多い系統。RVC は Retrieval（faiss kNN）ベースだが、so-vits-svc は **コンテンツ表現抽出 + 話者埋め込み条件付け**で VITS 系デコーダに渡すという、構造としては「素直な」系統で、両者の対比軸として整理しておく価値が高い
- 元々は **歌声変換**（SVC: Singing Voice Conversion）寄りで知られているが、話し声変換にも使われており、配信用途でも候補に挙がる
- ホスト側（[w-okada/voice-changer](../specs/w-okada-voice-changer.md)）からも対応モデルとして名前が挙がっており、UI 経由で触る場合の責任分界整理は RVC と同じく必要
- 「so-vits-svc」という名前自体が **元論文・元公式リポジトリ・派生フォーク（svc-develop-team/so-vits-svc, 4.0 / 4.1 系統等）** で複数の指し方がされており、責任分界が曖昧になりやすい
- 学習データの収集・配布チェックポイントの利用条件にライセンス的な敏感さがある（[CLAUDE.md](../../CLAUDE.md) 方針との接点）

ため、ホスト側の調査と独立して **モデル/アルゴリズム側の整理**を 1 ファイルに集約しておく必要がある。

このドキュメントが揃うと、

- RVC（Retrieval ベース系）と so-vits-svc（コンテンツ表現 + 話者条件付け系）の対比が **構造レベル**で書けるようになる（後続の Beatrice / MMVC 等の比較にも転用できる）
- w-okada/voice-changer 経由で so-vits-svc を触る際に、UI に出てくるピッチシフト / F0 抽出方式等のパラメータを**モデル側の知識**として裏付けられる
- so-vits-svc 系のチェックポイントを扱う際の**ライセンス判断軸**を、RVC とは独立した軸で明文化できる（公式リポジトリの License・「歌声」素材という性質ゆえの追加リスク）

## スコープ

含める:

- **公式系リポジトリ**（`svc-develop-team/so-vits-svc` を主軸として Phase 1 で確定。アーカイブ済みリポジトリや派生フォークとの位置関係も合わせて整理）で公式に説明されている範囲での
  - 全体アーキテクチャ（**コンテンツ表現抽出器（SoftVC / ContentVec / HuBERT 系）+ 話者埋め込み + F0 条件 + VITS ベースのデコーダ** という大枠）
  - **4.0 / 4.1 等の主要バージョン**の差分（README で明示されている範囲）
  - 学習プロセスの大枠（必要な学習データ量・前処理ステップ・F0 抽出 / 話者埋め込み構築の段階）
  - 推論パラメータと UI 露出されているもの（F0 抽出方式の選択肢、ピッチシフト、クラスタリング / 話者埋め込み関連のスイッチ等）
  - 配布形態とライセンス（リポジトリのライセンス、配布される事前学習モデル / Vocoder の利用条件、歌声素材という性質に起因する追加注意点）
- [voice-changer-types §4](../specs/voice-changer-types.md#4-配信用途で見るべき主な評価軸) 評価軸への暫定マッピング（モデル単体として答えられる項目に絞る）
- **w-okada/voice-changer 側で既に整理された内容との対応関係**
  - [w-okada/voice-changer §4.2](../specs/w-okada-voice-changer.md#42-モデル固有のパラメータ) で挙げた so-vits-svc 固有パラメータと、so-vits-svc 本家側で語られているパラメータの対応
  - ホストで吸収される部分 / モデル本来の挙動 の責任分界の再確認
- **RVC との対比**（[rvc.md](../specs/rvc.md) との突き合わせ）
  - 構造上の違い: Retrieval（faiss kNN）ベース vs コンテンツ表現 + 話者埋め込み条件付け
  - 入出力サンプリングレート・F0 抽出方式の選択肢・推論パラメータの共通点/差異
  - 配布チェックポイントの運用感の違い（特に歌声素材という性質に起因する違い）

含めない:

- 実測のレイテンシ / 音質 / GPU 使用量（自分で計測したものは別途 `experiments/` 配下に記録）
- **第三者の声を許諾なく再現するような利用方法**および**特定のキャラクター音声 / 歌唱モデルへのリンク・取得手順**（[CLAUDE.md](../../CLAUDE.md) 方針）
- インストール手順そのものの完全な再掲（必要になった時点で `experiments/` 側に書く）
- 学習スクリプトのコード詳細（ソース読解レベル）— 必要になった時点で別タスク化
- so-vits-svc-fork / DDSP-SVC / Diff-SVC など派生プロジェクトの個別深掘り（名前と位置関係は触れるが、深掘りは別タスク）

## 対応方針

成果物は `docs/specs/so-vits-svc.md` の 1 ファイル。
冒頭に voice-changer-types での位置付けを 1 行で書く（例: 「技術アプローチ: 1.3 ニューラル変換（コンテンツ表現 + 話者条件付け系）/ 提供形態: 2.1 ローカル OSS / モデル形態: 3.1 話者依存」）。

構成案（[rvc.md](../specs/rvc.md) の構成と並行できる形にする）:

1. 概要（so-vits-svc とは何か / 名前が指すレイヤの整理: アルゴリズム / 公式系リポジトリ / 派生フォーク / 配布チェックポイント）
2. アーキテクチャ概観（**コンテンツ表現抽出 / 話者埋め込み / F0 条件 / VITS デコーダ** の構成、4.0 / 4.1 等の差分）
3. 学習プロセスの大枠（必要な学習データ・F0 抽出・話者埋め込み・チェックポイント生成）
4. 推論パラメータ（F0 抽出方式、ピッチシフト、クラスタリング / 話者埋め込み関連のスイッチ等。w-okada 側 UI 露出との対応表）
5. 配布形態とライセンス（リポジトリ本体・事前学習モデル・Vocoder 同梱・コミュニティ配布チェックポイントの扱い、歌声素材という性質に起因する追加リスク）
6. w-okada/voice-changer 経由で触る場合の責任分界（ホスト側で吸収される / モデル側で決まる部分の対応関係）
7. **RVC との対比**（構造・パラメータ・運用面の差をまとめる）
8. voice-changer-types §4 評価軸への暫定マッピング（実測前提の項目は「未計測」と明記）
9. 未確認事項 / 後続タスク

記述ルール（[rvc.md](../specs/rvc.md) と統一）:

- **公式リポジトリ / 公式 README** に書かれていることと、コミュニティで一般に言われていることを明確に分ける
- 実測値・断定的な性能評価は書かない（[CLAUDE.md](../../CLAUDE.md) 方針）
- 「アルゴリズム so-vits-svc として共通」と「公式 WebUI / 派生フォーク / w-okada/voice-changer など実装側の話」を混ぜない
- 学習データ・配布チェックポイントのライセンスは、リポジトリ本体ライセンスと**分けて**書く
- 第三者キャラクター音声・歌唱モデルのチェックポイント取得手順は書かない（言及はしても、リンク・手順は記載しない）
- 「RVC との比較」を書くときは、自分達のラボでの実測ではなく **README / 公式に書かれている特性ベースの構造比較** に留める
