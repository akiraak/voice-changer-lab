# 調査プラン: 仮想オーディオデバイスまとめ

## 目的・背景

[voice-changer-types](../specs/voice-changer-types.md) §2 補助カテゴリ「周辺ツール」では、声変換そのものではないが運用上ほぼ必ず組み合わせる層として**仮想オーディオデバイス**を挙げ、代表例として `VB-CABLE / VoiceMeeter Banana / Potato / BlackHole / Soundflower (旧 macOS)` を名前出ししたまま止まっている。

これまでの調査は OSS リアルタイム声変換 4 種（[rvc.md](../specs/rvc.md) / [so-vits-svc.md](../specs/so-vits-svc.md) / [beatrice.md](../specs/beatrice.md) / [w-okada-voice-changer.md](../specs/w-okada-voice-changer.md)）+ クラウド SaaS 棚卸し 1 件（[cloud-saas-realtime.md](../specs/cloud-saas-realtime.md)）に偏っており、**「声変換エンジンの出力を配信ソフトへ渡す経路」側の解像度が空白**になっている。リアルタイム声変換の評価軸は声変換エンジン単体ではなく `マイク → (ノイズ抑制) → 声変換 → 仮想デバイス → OBS / 通話ソフト` の**経路全体のレイテンシ・取り回し**で決まるため、仮想オーディオデバイス側が空白のままだと:

- [voice-changer-types §4 配信用途で見るべき主な評価軸](../specs/voice-changer-types.md#4-配信用途で見るべき主な評価軸) の「入出力構成」「エコシステム」軸が評価できない
- [w-okada-voice-changer.md §6 配信ソフトとの連携](../specs/w-okada-voice-changer.md) と、後続の [experiments/w-okada-voice-changer/](../../experiments/w-okada-voice-changer/) 実測テンプレートで「どの仮想デバイスでどう繋ぐか」を都度考えることになり、計測条件が揃わない
- [cloud-saas-realtime.md §4.1](../specs/cloud-saas-realtime.md#41-判定サマリ足切り判断) で Voice.ai が "VB-Audio Virtual Cable 経由で OS に仮想入力デバイスとして露出 / Streamlabs OBS 互換を公式明記" を理由に判定 A になっているが、その前提となる VB-CABLE 側の仕様（モノラル/ステレオ / サンプルレート / ライセンス / ループバック構成）がこちら側の spec で裏取りされていない

このタスクは、**配信周辺ツール調査の第 1 弾**として、上記の空白を埋めることを目的とする。同シリーズ第 2 弾の `OBS Studio との接続パターンまとめ` は本書の成果物を前提にする想定のため、本書は OBS 側の経路図には踏み込まず、**仮想オーディオデバイス単体の仕様棚卸し**に絞る。

このドキュメントが揃うと、

- voice-changer-types §2 補助カテゴリ・§4 評価軸の「入出力構成」が、ツール単位の仕様棚卸しと**ラベル付き比較表**で参照できるようになる
- w-okada / 他 OSS 声変換ツールの実測時、`experiments/<tool>/<run>.md` で「仮想デバイスは何を使ったか」を**選択肢から指定する**形に統一できる
- クラウド SaaS（Voice.ai 等）が "仮想ケーブル経由で配信ソフトへ" と謳う構成の**前提となる仮想ケーブル側の制約**（モノラル/サンプリングレート/ライセンス）が裏取り済みの状態になり、SaaS 棚卸しの判定根拠が補強される
- OBS Studio との接続パターン調査（次タスク）の前提が固まる

## スコープ

含める:

- 棚卸し対象（Phase 1 で確定する。少なくとも以下を起点とする）
  - **VB-CABLE**（VB-Audio。Windows 向け仮想オーディオケーブルの定番）
  - **VoiceMeeter Banana / Potato**（VB-Audio。仮想ミキサー兼仮想オーディオデバイス）
  - **BlackHole**（macOS 向け仮想オーディオドライバ。Existential Audio 提供）
  - 補足扱い: **Soundflower**（macOS 旧。BlackHole の前世代として位置付け確認のみ）、**Linux/PulseAudio・PipeWire の仮想シンク**（参考。本ラボのメイン環境ではないが他資料との照合のために 1 段落で触れる）
- 各ツールについて、**公式サイト / 公式ドキュメント / ダウンロードページ / 公式ライセンス記述**で確認できる範囲での
  - 提供形態（無償 / Donationware / 有償 / オープンソース）と**ライセンス / 商用配信での利用可否の有無**（明示されていれば書き、書かれていなければ「公式記述なし」と明示）
  - 動作プラットフォーム（Windows のバージョン / macOS のバージョン / Apple Silicon 対応の有無 / ARM Windows 対応の有無）
  - 仕組みの粒度（仮想入出力ペア 1 本 / 複数ペア / 仮想ミキサー併設 / ループバック / マルチクライアント対応 など、公式が説明する範囲）
  - サンプルレート / ビット深度 / チャンネル数の公式仕様（公称値のみ。実測は含めない）
  - インストール / アンインストールの**手順の所在**（公式ガイドへのリンクと、想定される注意点を 1〜2 行）
  - 配信ソフト・声変換ホストから見たときの**OS 上のデバイス名**（公式ドキュメントが明示している場合）
  - **既知の制約 / ハマりどころ**として公式 FAQ / Known Issues に書かれているもの
- 配信用途で組み合わせる典型構成（**経路図レベル**で、`マイク → 声変換 → 仮想ケーブル → OBS` 等の代表 3〜5 パターン。OBS 側の詳細設定は別タスクのスコープなので、ここでは「仮想ケーブルがどこに入るか」までの粒度）
- voice-changer-types §2 補助カテゴリ / §4 評価軸への反映方針
- w-okada-voice-changer.md §6 配信ソフトとの連携 / `experiments/w-okada-voice-changer/_template.md` への反映方針（「使った仮想デバイス」を本書の選択肢で書ける形にする差分提案）
- 後続タスク起票（個別ツールで深掘りが必要なもの、OBS Studio 接続パターン側で前提にすべき項目）

含めない:

- **実測のレイテンシ / 音質 / CPU 負荷**（[CLAUDE.md](../../CLAUDE.md) 方針。公称値の引用はベンダー名・出典付きで明示し、本ラボでの実測値とは混ぜない）
- **OBS Studio 側の設定詳細・モニタリング経路設計**（次タスク `OBS Studio との接続パターンまとめ` のスコープ。本書では「仮想ケーブルがどこに刺さるか」までで止める）
- **ノイズ抑制 / ゲート系プラグイン**（[voice-changer-types §2 補助カテゴリ](../specs/voice-changer-types.md#補助カテゴリ-周辺ツール) のうち別レイヤー。本書のスコープ外）
- **通話アプリ（Discord 等）固有の仮想デバイス取り回し**（必要になった時点で個別タスクへ。本書ではあくまで「配信ソフトに渡すための仮想デバイス」までを扱う）
- 棚卸し対象に**周辺ツールを片っ端から追加すること**（Phase 1 で対象集合を確定し、本書は確定集合に集中する。スコープ外の有力候補は §未確認事項 / 後続タスクに名前だけ残す）
- 著作権上問題のある配布物（クラックされたドライバなど）の取得手順 — そもそも触れない

## 対応方針

成果物は `docs/specs/virtual-audio-devices.md` の 1 ファイル。
冒頭で本書が [voice-changer-types §2 補助カテゴリ「周辺ツール」](../specs/voice-changer-types.md#補助カテゴリ-周辺ツール) のどこを掘り下げるか（仮想オーディオデバイス層のみ。ノイズ抑制 / 配信ソフト本体は別）を 1 行で示す。個別ツール spec（[rvc.md](../specs/rvc.md) / [so-vits-svc.md](../specs/so-vits-svc.md) / [beatrice.md](../specs/beatrice.md) / [w-okada-voice-changer.md](../specs/w-okada-voice-changer.md)）とは異なり、**1 ツール深掘りではなく複数ツールの横断棚卸し**である点を、[cloud-saas-realtime.md](../specs/cloud-saas-realtime.md) と同じ立て付けで明示する。

構成案:

1. 概要（本書の位置付け / 対象集合 / 「仮想オーディオデバイス」と本書で呼ぶものの範囲定義 — ドライバレベルの仮想デバイスと、ユーザーランドのソフトウェアミキサーをどう分けるか）
2. 評価項目一覧（後述の棚卸し表のカラム定義。`提供形態・ライセンス / 動作プラットフォーム / 仮想 IO ペア構成 / サンプルレート・チャンネル数 / 配信ソフトから見たデバイス名 / 既知の制約` を縦軸のツール × 横軸の項目で表す）
3. ツール別エントリ（VB-CABLE / VoiceMeeter Banana / Potato / BlackHole / Soundflower / Linux 仮想シンク）。各エントリは
   - 公式情報源へのリンク（公式サイト / ダウンロードページ / ライセンスページ / FAQ）
   - 製品の**公式説明の引用範囲**（ベンダーが何と呼んでいるか・どの用途を公式に想定しているか）
   - 提供形態（無償 / Donationware / 有償）とライセンス（特に商用配信での利用可否について公式が触れているか）
   - 動作プラットフォーム（OS 別の対応状況。Apple Silicon / Windows ARM の対応有無）
   - 仮想 IO ペア構成（何本のケーブル / マルチクライアント / ループバックの有無 / ミキサー機能の有無）
   - サンプルレート / ビット深度 / チャンネル数の公式仕様
   - 配信ソフト・声変換ホストから見たときの OS 上のデバイス名（公式記述がある場合のみ）
   - インストール / アンインストールの手順所在（公式ガイドへのリンクと注意点）
   - 既知の制約 / 公式 FAQ に書かれているハマりどころ
   - 未確認事項（公式ドキュメントから読み取れなかった項目）
4. 横断棚卸し表（§2 で定義した評価項目のマトリクス。配信用途で**そもそも候補に入るか / 入らないか**の足切り判断が一目で分かる形。OS × ツールの組合せで「Windows 配信なら VB-CABLE / VoiceMeeter」「macOS 配信なら BlackHole」のような早見表にする）
5. 配信用途での典型構成（経路図レベルで代表 3〜5 パターン。`マイク → 声変換ホスト (w-okada) → 仮想ケーブル → OBS` を中心に、ループバック・モニター分岐などのバリエーション。OBS 内部設定には踏み込まない）
6. voice-changer-types / w-okada-voice-changer / experiments テンプレへの反映方針（具体的な更新差分の案までを書く）
7. 未確認事項 / 後続タスク（個別ツールの深掘り、OBS 接続パターン側で本書を前提にすべき項目、通話アプリ向け取り回し、ノイズ抑制レイヤーの追加調査など）

記述ルール（[cloud-saas-realtime.md](../specs/cloud-saas-realtime.md) / [rvc.md](../specs/rvc.md) / [so-vits-svc.md](../specs/so-vits-svc.md) / [beatrice.md](../specs/beatrice.md) / [w-okada-voice-changer.md](../specs/w-okada-voice-changer.md) と統一）:

- **公式サイト / 公式ドキュメント / 公式 FAQ**に書かれていることと、コミュニティで一般に言われていることを明確に分ける。出典を行ごとに明記する
- 実測値・断定的な性能評価は書かない（[CLAUDE.md](../../CLAUDE.md) 方針）。**ベンダー公称レイテンシや CPU 負荷は「ベンダー公称」として明示**し、本ラボでの実測値とは混ぜない
- 製品名 / デバイス名は**ベンダーがそう呼んでいる名称**をそのまま書く（OS のデバイス一覧での見え方は、公式が明示している場合のみ採用し、推測では書かない）
- ライセンス / 商用利用の可否は**ベンダー公式の明示がある項目のみ書く**。明示がない場合は「公式記述なし」と明記し、勝手に判断しない
- ツールごとに**最終確認日**を明記する（バイナリ配布のバージョンと OS 対応は変動が早いため）
- 価格表記は**ベンダー表記そのまま**（USD / EUR / JPY / Donation 金額レンジ等）。為替換算は行わない
