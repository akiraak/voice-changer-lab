# 調査プラン: w-okada/voice-changer の仕様

## 目的・背景

[voice-changer-types](../specs/voice-changer-types.md) §1.4 ハイブリッド / §2.1 ローカル OSS に位置付けた w-okada/voice-changer を、個別ツールとして掘り下げる。
本ツールは単体の声質変換アルゴリズムではなく、**RVC / so-vits-svc / Beatrice / MMVC / DDSP-SVC 等の複数モデルを切り替えて動かす実行ホスト**として紹介されることが多い。
配信ラボとしてこのリポジトリの上で他のモデル（RVC / so-vits-svc 等）の個別調査を進めるうえで、まず「ホスト側がどこまで面倒を見てくれるか」「ユーザーが意識すべき責任分界点はどこか」を整理しておく必要がある。

このドキュメントが揃うと、

- 後続の RVC / so-vits-svc / Beatrice の個別調査時に「モデル本体の話」と「ホスト経由で出てくる挙動」を切り分けて書ける
- 配信用途で OBS / 仮想オーディオデバイスと接続する際の IO 構成検討の足場ができる
- どの調査項目が「ホストに聞けば分かる」「モデル側を見ないと分からない」「自分で実測しないと分からない」かが明確になる

## スコープ

含める:

- 公式リポジトリ（w-okada/voice-changer）で公式に説明されている範囲での
  - 全体アーキテクチャ（クライアント / サーバ構成、Web UI、Python 側、ネイティブ実行など）
  - 対応モデル一覧と、モデルごとに「ホスト側でできること」（モデル差し替え、パラメータ調整など）の範囲
  - 前処理 / 後処理 として提供されるもの（ノイズ抑制、音量正規化、クロスフェード、ピッチ調整 など、公式に語られている項目）
  - 音声 IO の構成（入力デバイス / 出力デバイス / 仮想オーディオデバイス経由 / ブラウザマイク経由 など、公式に言及されている経路）
  - 配布形態とライセンス（リポジトリのライセンス、配布バイナリの利用条件、同梱モデル/サンプルの取り扱い）
- voice-changer-types §4 の評価軸（レイテンシ / 必要 GPU / 入出力構成 / カスタマイズ性 / 配信での利用可否 / エコシステム）への暫定マッピング

含めない:

- 実測のレイテンシ / 音質 / GPU 使用量（自分で計測したものは別途 `experiments/` 配下に記録）
- 個別モデル（RVC / so-vits-svc 等）そのものの仕様詳細（それぞれの個別調査タスクで扱う）
- 第三者の声を許諾なく再現するような利用方法（プロジェクト方針外）
- インストール手順そのものの完全な再掲（必要になった時点で `experiments/` 側に書く）

## 対応方針

成果物は `docs/specs/w-okada-voice-changer.md` の 1 ファイル。
冒頭に voice-changer-types での位置付けを 1 行で書く（例: 「技術アプローチ: 1.4 ハイブリッド / 提供形態: 2.1 ローカル OSS / モデル形態: ロードするモデル依存」）。

構成案:

1. 概要（プロジェクトとしての立ち位置・「実行ホスト」としての意味）
2. アーキテクチャ概観（クライアント / サーバ、起動形態、依存スタック）
3. 対応モデル一覧と切替の仕組み（モデルごとの差異 / 共通化されている部分）
4. 前処理・後処理として提供されているもの
5. IO 構成（入出力デバイス、仮想オーディオデバイス、ブラウザ経由などのパターン）
6. 配信ソフト（OBS 等）との接続パターンの公式言及
7. ライセンス / 配布物の利用条件と、本ラボで扱う際の注意点
8. voice-changer-types §4 評価軸への暫定マッピング（実測前提の項目は「未計測」と明記）
9. 未確認事項 / 後続タスク

記述ルール:

- 公式リポジトリ / 公式 README / 公式 Wiki に書かれていることと、一般に言われていることを明確に分ける
- 実測値・断定的な性能評価は書かない（[CLAUDE.md](../../CLAUDE.md) 方針）
- 「ホスト側で吸収される挙動」と「モデル側の挙動」を混ぜない。混ざりやすい項目は両方を 1 セットで書く
- ライセンスについては、リポジトリのライセンスと、同梱・配布されるモデル/サンプル音声の利用条件を**分けて**書く
