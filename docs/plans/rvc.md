# 調査プラン: RVC の仕様

## 目的・背景

[voice-changer-types](../specs/voice-changer-types.md) §1.3 ニューラル変換（Retrieval ベース系）/ §2.1 ローカル OSS / §3.1 話者依存型 に位置付けた **RVC (Retrieval-based Voice Conversion)** を、個別ツールとして掘り下げる。

RVC は本ラボで現時点で最も「実運用例が多く、配布されているチェックポイントも豊富」な系統だが、

- ホスト経由（[w-okada/voice-changer](../specs/w-okada-voice-changer.md)）で間接的にしか触れていない
- 「RVC」という名前が **アルゴリズム名 / 公式リポジトリ名 / 配布されているチェックポイント** の 3 つを指すことが多く、責任分界が曖昧になりがち
- 学習データの収集・チェックポイントの利用条件にライセンス的な敏感さがある（[CLAUDE.md](../../CLAUDE.md) 方針との関係が強い）

ため、ホスト側の調査と独立して **モデル/アルゴリズム側の整理**を 1 ファイルに集約しておく必要がある。

このドキュメントが揃うと、

- 後続の so-vits-svc / Beatrice 等の比較時に「RVC は Retrieval ベースで、コンテンツ表現は HuBERT 系で…」のような対比を簡潔に書ける
- w-okada/voice-changer 経由で RVC を触る際に、UI に出てくる F0 抽出方式 / INDEX 比率 / TUNE 等の挙動を**モデル側の知識**として裏付けられる
- RVC モデルチェックポイントを扱う際の**ライセンス判断軸**を明文化しておける（学習データ由来の規約・第三者キャラクター音声の扱い）

## スコープ

含める:

- **公式リポジトリ**（`RVC-Project/Retrieval-based-Voice-Conversion-WebUI` を想定。実際の現行リポジトリ名は Phase 1 で確認）で公式に説明されている範囲での
  - 全体アーキテクチャ（コンテンツ表現抽出器 + Retrieval（kNN / faiss）+ デコーダ という大枠）
  - **v1 / v2** モデルの差分（README で明示されている範囲）
  - 学習プロセスの大枠（必要な学習データ量・前処理ステップ・f0 抽出 / index 構築の段階）
  - 推論パラメータと UI 露出されているもの（F0 抽出方式の選択肢、INDEX 比率、ピッチシフト など）
  - 配布形態とライセンス（リポジトリのライセンス、配布される事前学習モデル / Vocoder の利用条件）
- [voice-changer-types §4](../specs/voice-changer-types.md#4-配信用途で見るべき主な評価軸) 評価軸への暫定マッピング（モデル単体として答えられる項目に絞る）
- **w-okada/voice-changer 側で既に整理された内容との対応関係**
  - [w-okada/voice-changer §4.2](../specs/w-okada-voice-changer.md#42-モデル固有のパラメータ) で挙げた RVC 固有パラメータ（F0 Det / INDEX）と、RVC 本家側で語られているパラメータの対応
  - ホストで吸収される部分 / モデル本来の挙動 の責任分界の再確認

含めない:

- 実測のレイテンシ / 音質 / GPU 使用量（自分で計測したものは別途 `experiments/` 配下に記録）
- **第三者の声を許諾なく再現するような利用方法**および**特定のキャラクター音声モデルへのリンク・取得手順**（[CLAUDE.md](../../CLAUDE.md) 方針）
- インストール手順そのものの完全な再掲（必要になった時点で `experiments/` 側に書く）
- 学習スクリプトのコード詳細（ソース読解）— 必要になった時点で別タスク化

## 対応方針

成果物は `docs/specs/rvc.md` の 1 ファイル。
冒頭に voice-changer-types での位置付けを 1 行で書く（例: 「技術アプローチ: 1.3 ニューラル変換（Retrieval ベース系）/ 提供形態: 2.1 ローカル OSS / モデル形態: 3.1 話者依存」）。

構成案:

1. 概要（RVC とは何か / 名前が指す 3 つのレイヤ「アルゴリズム / 公式リポジトリ / 配布チェックポイント」の整理）
2. アーキテクチャ概観（コンテンツ表現抽出 / Retrieval (faiss index) / デコーダの 3 段構成、v1 と v2 の差分）
3. 学習プロセスの大枠（必要な学習データ・f0 抽出・index 構築・チェックポイント生成）
4. 推論パラメータ（F0 抽出方式、INDEX 比率、ピッチシフト等。w-okada 側 UI 露出との対応表を作る）
5. 配布形態とライセンス（リポジトリ本体・事前学習モデル・Vocoder 同梱・コミュニティ配布チェックポイントの扱い）
6. w-okada/voice-changer 経由で触る場合の責任分界（ホスト側で吸収される / モデル側で決まる部分の対応関係）
7. voice-changer-types §4 評価軸への暫定マッピング（実測前提の項目は「未計測」と明記）
8. 未確認事項 / 後続タスク

記述ルール:

- **公式リポジトリ / 公式 README** に書かれていることと、コミュニティで一般に言われていることを明確に分ける
- 実測値・断定的な性能評価は書かない（[CLAUDE.md](../../CLAUDE.md) 方針）
- 「アルゴリズム RVC として共通」と「公式 WebUI / w-okada/voice-changer など実装側の話」を混ぜない
- 学習データ・配布チェックポイントのライセンスは、リポジトリ本体ライセンスと**分けて**書く
- 第三者キャラクター音声のチェックポイント取得手順は書かない（言及はしても、リンク・手順は記載しない）
