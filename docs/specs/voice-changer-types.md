# ボイスチェンジャーの種類

リアルタイム声変換ツールを後から個別比較するための**見取り図**。
個別ツールの遅延・音質・GPU 要件などは扱わず、「どんな軸で分類できるか」「各軸でどんなタイプがあるか」までをまとめる。

> このドキュメントは一般に知られている特性のみを記述する。実測値・断定的な性能評価は含めない（[CLAUDE.md](../../CLAUDE.md) の方針）。
> Phase 2 までで、分類軸のドラフトと各分類への代表ツールの当てはめ（名前出しのみ）を反映している。各ツールの遅延・音質・GPU 要件などの実測比較や個別深掘りは含まない（後続タスク）。

## 全体像

ボイスチェンジャーは大きく 3 つの軸で位置付けられる。

- **技術アプローチ**: どうやって声を変えるか（信号処理 / ニューラル変換 / ハイブリッド など）
- **提供形態**: どこで動かすか・どう手に入れるか（ローカル OSS / ローカル商用 / クラウド SaaS）
- **モデル形態**: ターゲット声の決め方（話者依存 / 多話者 / Zero-shot / プリセット）

実際のツールはこの 3 軸の組合せで表現できる（例: 「ニューラル変換 × ローカル OSS × 話者依存」）。

```
                   ┌─ 技術アプローチ ─┐
                   │  - 信号処理       │
                   │  - ボコーダ系     │
                   │  - ニューラル変換 │
                   │  - ハイブリッド   │
                   └───────────────────┘
                            │
   ┌─ 提供形態 ─┐           │           ┌─ モデル形態 ──────────┐
   │ - OSS      │ ────────  ◇  ──────── │ - 話者依存            │
   │ - 商用     │                        │ - 多話者 / 非依存     │
   │ - SaaS     │                        │ - Zero-shot           │
   └────────────┘                        │ - プリセット固定      │
                                         └───────────────────────┘
```

## 1. 技術アプローチによる分類

「入力音声をどう加工してターゲットの声に近づけるか」の方式区分。

### 1.1 信号処理ベース (DSP)

- 古典的なピッチシフト・フォルマント変換・EQ などを組み合わせる方式
- 音声を抽象的な特徴量に落とさず、波形/スペクトルを直接操作する
- 一般に低レイテンシで CPU だけでも動くが、別人の声に「化ける」用途には向かないとされる
- 配信用途では「声色のちょい変え」やキャラ加工レイヤーの 1 段として登場することが多い

**代表例（信号処理レイヤーとして登場するもの）:** 各種 DAW 向けピッチシフト / フォルマント変換プラグイン全般、Voicemod / Voidol などの製品に内蔵された DSP エフェクト層、VST/AU プラグイン形式のリアルタイムボイスチェンジャー。単体の「信号処理オンリー製品」というより、後述 §1.4 ハイブリッドの一段として登場することが多い。

### 1.2 ボコーダ系 / 古典的声質変換

- ソース音声を励振 + 声道特徴に分解し、フィルタ部分を入れ替えるタイプ
- WORLD などのアナリシス/シンセシスを利用する古典 VC もここに含まれる
- 統計モデル（GMM など）で写像を学習する手法もここに分類する
- ニューラル方式が登場する前の VC 研究の主流系統

**代表例:** WORLD ボコーダを用いた古典的声質変換実装、GMM-VC / DNN-VC など研究系コード（製品として配信向けに広く流通している例は限定的で、現代では §1.3 ニューラル変換が主流）。

### 1.3 ニューラル変換 (Neural Voice Conversion)

入力波形 → 中間表現 → ターゲット話者の波形、をニューラルネットで学習する系統。
内部で使われるアーキテクチャや中間表現で更にサブカテゴリに分かれる。

- **コンテンツ表現 + 話者条件付け系**: 入力から音韻 / コンテンツ表現を取り出し、ターゲット話者の条件を与えて合成する。VITS / so-vits-svc 系などが該当する系統
  - 代表例: [so-vits-svc](./so-vits-svc.md)（SoftVC 系コンテンツエンコーダ + VITS デコーダ + NSF-HiFiGAN）、[Beatrice](./beatrice.md)（PhoneExtractor + PitchEstimator + WaveGenerator、内部 vector quantization を持つが外部 retrieval は持たない）、MMVC, FreeVC, VITS-SVC 系派生（MMVC / FreeVC 等の内部実装は個別確認が必要だが、so-vits-svc・Beatrice の系統と同じ「コンテンツ抽出 + 話者条件で合成」の括りでこの位置に置く）
- **Retrieval ベース系**: 学習したターゲット話者の埋め込みを kNN 等で参照して特徴量を置換する系統
  - 代表例: RVC（Retrieval-based Voice Conversion。名称そのものがこの系統を示す）
- **拡散モデル (Diffusion) 系**: 拡散ベースのデコーダや声質変換モデルを使う系統
  - 代表例: Diff-SVC, DiffVC 系研究実装（リアルタイム動作を狙った派生がいくつか存在する）
- **End-to-end 波形変換系**: 波形 → 波形を 1 本のネットで学習する系統
  - 代表例: AutoVC, CycleGAN-VC 系（主に研究実装。配信ツールとして直に使われる形では出回りにくい）

### 1.4 ハイブリッド

- 信号処理での前処理（ノイズ抑制 / 音量正規化）と、ニューラル変換を組み合わせる構成
- ニューラル変換の出力にエフェクト（リバーブ / EQ）を後段でかける構成
- 「ボイスチェンジャー製品」と呼ばれるものは、純粋な単一方式ではなく、ほぼハイブリッドで提供される

**代表例:**

- w-okada/voice-changer: 単体のアルゴリズムというより RVC / so-vits-svc / Beatrice / MMVC / DDSP-SVC 等の各モデルを切り替えて動かせる**実行ホスト**。前処理 / 後処理 / IO 配線も担うため、運用上は §1.4 ハイブリッドに位置する
- DDSP-SVC: 名前のとおり DSP 要素とニューラルを組み合わせた歌声変換系
- Voicemod / Voidol などの商用デスクトップ製品: ニューラル変換 + 信号処理エフェクト + プリセット管理を一体化して提供（内部実装はブラックボックスなので「ハイブリッド構成として提供されている」までを記述）

## 2. 提供形態による分類

「どこで動くか / どう手に入れるか」の区分。配信用途で**そもそも候補に入るか**の足切りに使える軸。

### 2.1 ローカル OSS

- ソースコードが公開されており、自分のマシンで動かす
- GPU を使う前提のものが多い
- モデルの差し替え・自前学習・他ツールとの連携の自由度が高い
- 一方で、セットアップ・依存関係・モデル配布元のライセンスを自分で面倒を見る必要がある

**代表例:** RVC, so-vits-svc, w-okada/voice-changer, Beatrice, MMVC, DDSP-SVC（各リポジトリのライセンス・配布モデルの利用条件は個別確認。「コードが OSS」と「配布される学習済みモデルが自由に使えるか」は別問題なので、ツールごとの調査時に必ず分けて確認する）。

### 2.2 ローカル商用ソフトウェア

- 買い切り / サブスクで配布される、インストール型のデスクトップアプリ
- GUI・サポート・あらかじめ用意されたプリセット声が提供されることが多い
- 内部実装はブラックボックス（信号処理寄り / ニューラル寄り は製品次第）
- ライセンス・配信での利用範囲は製品ごとに規約確認が必要

**代表例:** Voicemod, Voidol（クリムゾンテクノロジー）, Voice.ai のデスクトップクライアント等。配信での商用利用可否・収益化可否は製品ごとに規約が異なるため、採用前に必ず確認する。

### 2.3 クラウド SaaS / API

- 変換処理をクラウド側で行い、結果をストリームで返す
- ローカル GPU が不要だが、ネットワーク経由のため遅延・通信品質・利用料が論点
- 配信のリアルタイム入力に乗せられるかは SaaS ごとの提供形態次第（一般には会話/通話向け SDK の有無で判断する）

**代表例とリアルタイム配信経路の有無:** 棚卸しの一次判断は [cloud-saas-realtime.md §4.1](./cloud-saas-realtime.md#41-判定サマリ足切り判断) を参照（最終確認日 2026-05-12）。

- **CoeFont** — 判定 **E（未確認）**。VC LP は "Real-time" を標榜するが、連続マイク入力 / ストリーム出力 / 仮想オーディオ経路の公式技術仕様および ToS 本文が未確認
- **Resemble AI** — 判定 **C（部分的）**。リアルタイム TTS は WebSocket ストリーミング（200ms TTFS 公称）で揃うが、本書スコープのリアルタイム声変換（Speech-to-Speech）は同期 REST / 最大 5 分 WAV のファイル入力で連続マイク入力 / ストリーム出力非充足
- **ElevenLabs** — 判定 **C（部分的）**。`speech-to-speech/stream` は応答 SSE のみで、入力はファイル単位（1 セグメント最大 5 分）。公式デスクトップ / OBS プラグイン / 仮想オーディオ連携の公式言及なし
- **Voice.ai** — 判定 **A（公式リアルタイム配信経路あり）**。デスクトップアプリ "Free Real-Time AI Voice Changer for PC & Mac" が VB-Audio Virtual Cable 経由で OS に仮想入力デバイスとして露出。"Compatible with Streamlabs OBS" を公式明記。商用配信は Starter（$5/月）以上の commercial license プランが ToS 上の前提

棚卸し対象外の SaaS（Respeecher / Altered AI / Supertone / Voicemod クラウド / Murf.ai / PlayHT / Speechify / Camb.ai 等）は [cloud-saas-realtime.md §7.3](./cloud-saas-realtime.md#73-棚卸し対象に追加する候補-saas本書-phase-1-ではスコープ外) に名前のみ残し、必要が出た時点で個別タスクへ昇格する。

### 補助カテゴリ: 周辺ツール

声変換そのものではないが、運用上ほぼ必ず組み合わせる層。

- 仮想オーディオデバイス（変換後の音声を配信ソフトに渡すための仮想ケーブル）
  - 代表例: VB-CABLE, VoiceMeeter Banana / Potato, BlackHole (macOS)。横断棚卸しは [virtual-audio-devices.md](./virtual-audio-devices.md)（最終確認日 2026-05-12、§4 横断棚卸し表 / §5 配信用途の典型構成 P1〜P4 を参照）
  - Soundflower は macOS Big Sur 以前のレガシー扱いで、本ラボでの新規採用は非推奨（[virtual-audio-devices.md §3.5](./virtual-audio-devices.md#35-soundflower補足)）
- ノイズ抑制 / ゲート（変換前の入力をクリーンにする）
  - 代表例: NVIDIA Broadcast (旧 RTX Voice), Krisp, RNNoise
- 配信ソフト・通話ソフト（最終的な配信先）
  - 代表例: OBS Studio, Streamlabs Desktop、各種通話アプリ（Discord 等）

提供形態の軸ではこれらを「**周辺ツール**」として別立てし、声変換本体の比較とは混同しない。

## 3. モデル形態の軸

「ターゲットの声をどう決めるか」「ターゲットを増やすコスト」の軸。

### 3.1 話者依存型 (Target-specific)

- ターゲット話者ごとにモデル / 重みを学習する
- 1 つの声に対しては品質を高く追い込めるとされる
- 新しい声を増やすたびに学習データの収集と学習が必要
- OSS 系の声質変換は基本この形態が中心

**代表例:** RVC, so-vits-svc（基本的に 1 ターゲット話者 1 モデルを学習して切り替えるワークフロー）。

### 3.2 多話者 / 話者非依存型 (Many-to-many)

- 学習済みモデル 1 つで複数のターゲット話者を切り替えられる構成
- 話者埋め込み / 話者 ID を条件として渡す
- 各話者の品質は話者依存型ほど追い込めない場合もあるが、運用は楽になる

**代表例:** MMVC（Many-to-Many Voice Conversion）、多話者 VITS 系モデル、FreeVC。

### 3.3 Zero-shot / Any-to-any

- リファレンス音声を数秒〜数十秒与えるだけで、その声に変換する
- 個別の追加学習を必要としない（学習済みモデルだけで完結）
- 当然ながら**第三者の声を許諾なく真似る**用途への流用リスクが高く、利用規約・倫理面の判断が重い

**代表例:** OpenVoice, XTTS, FreeVC（Zero-shot 設定）、ElevenLabs / Resemble AI の Voice Cloning 系機能など。ただし広く流通しているのは TTS / バッチ変換用途が中心で、**配信向けに低遅延ストリームで動かせる Zero-shot VC の代表例は現時点では確立していない**（リアルタイム動作の可否は個別検証扱い）。

### 3.4 プリセット固定型

- ユーザーがモデルを学習・選択するのではなく、ベンダー側があらかじめ用意した「キャラ声」をそのまま使う
- 商用ボイスチェンジャー製品でよく見られる形態
- 中身は話者依存モデル群を切り替えていることが多いが、ユーザーから見ると「プリセットを選ぶだけ」の体験になる

**代表例:** Voicemod, Voidol などの商用デスクトップ製品が提供する「キャラ声 / プリセット」群。自前モデルの取り込みに対応するかは製品ごとに異なる。

## 4. 配信用途で見るべき主な評価軸

後続の個別調査タスクで揃えるべき観点をリストアップする（本ドキュメントでは値は埋めない）。

| 評価軸 | 何を見るか |
| --- | --- |
| レイテンシ | 入力 → 出力の片道遅延。配信での同期ズレ / 会話の自然さに影響 |
| 音質 | 自然さ・声の安定性・破綻のしにくさ。サンプルレートも明記 |
| 必要 GPU 性能 | VRAM 量 / 推奨世代。GPU 無しで動くか |
| CPU 負荷 | 配信ソフト・ゲームと併走させたときの余裕度 |
| 入出力構成 | 想定する入力デバイス・仮想オーディオデバイス経由の取り回し（仮想オーディオデバイスの選定軸は [virtual-audio-devices.md §4](./virtual-audio-devices.md#4-横断棚卸し表)、配信用途の典型構成は同 [§5](./virtual-audio-devices.md#5-配信用途での典型構成)） |
| モデル形態 | §3 のどれに該当するか。新しい声を増やすコスト |
| カスタマイズ性 | 自前モデルの学習可否・読み込み可否・ピッチや声色の調整幅 |
| 配信での利用可否 | 商用配信での利用可否、収益化可否、クレジット要件などライセンス上の制約 |
| エコシステム | 仮想オーディオデバイス・OBS との接続事例、コミュニティ規模、メンテ頻度（仮想オーディオデバイス側の対応状況は [virtual-audio-devices.md §4.2 OS × ツール早見表](./virtual-audio-devices.md#42-os--ツール早見表足切り判断)） |
| コスト | 初期費用 / サブスク / クラウド従量。GPU を持っていない場合の総額 |

## 5. このドキュメントの使い方

- 個別ツール調査ページ（`docs/specs/<tool-name>.md`）の冒頭で、本ドキュメントの分類のどこに該当するかを 1 行で書く
  - 例: 「技術アプローチ: 1.3 ニューラル変換 / 提供形態: 2.1 ローカル OSS / モデル形態: 3.1 話者依存」
- 候補ツールの棚卸し時に、**そもそも候補に入らない方式**（オフライン専用、配信不可ライセンス等）をこの軸で早期に除外する
- 各ツールの「遅延 ◯◯ ms」のような実測値は本ドキュメントには書かず、個別ページ / `experiments/` 配下に記録する

## 未確定事項 (Phase 3 以降で再評価)

- §1.3 の細分（コンテンツ表現系 / Retrieval 系 / Diffusion 系 / End-to-end 系）の境界が現実のツールでどこまでクリアに引けるかは個別ツール調査時に再評価する
- Phase 2 で名前を置いた MMVC / DDSP-SVC などの分類は「一般にそう呼ばれている系統」に沿って暫定置きしているだけで、各ツールの正確な内部実装は個別ページで裏取りする（**Beatrice は [beatrice.md](./beatrice.md) §2.1 で「コンテンツ表現 + 話者条件付け系、内部 VQ あり / 外部 retrieval なし」に確定済み**）
- §2.3 クラウド SaaS にリアルタイム入力を流し込むユースケースは、[cloud-saas-realtime.md](./cloud-saas-realtime.md) で対象 4 SaaS（CoeFont / Resemble AI / ElevenLabs / Voice.ai）の公式情報源を棚卸し済み（最終確認日 2026-05-12）。**現時点で「マイク入力 → 別声 → 配信ソフトに流せるストリーム出力」を公式機能として提供していると公式情報源で裏が取れたのは Voice.ai のデスクトップアプリ経由 1 件のみ**。Resemble AI / ElevenLabs はリアルタイム TTS は揃っているがリアルタイム声変換は揃わない。CoeFont は技術仕様自体が未確認で再裏取りが必要。後続候補（Respeecher / Altered AI / Supertone / Voicemod クラウド連携 等）は [cloud-saas-realtime.md §7.3](./cloud-saas-realtime.md#73-棚卸し対象に追加する候補-saas本書-phase-1-ではスコープ外) に列挙済み
- §3.3 Zero-shot は「配信向けに低遅延で動かせる代表例」がまだ確立していないため、後続の個別調査で更新する余地が大きい
