# ボイスチェンジャーの種類

リアルタイム声変換ツールを後から個別比較するための**見取り図**。
個別ツールの遅延・音質・GPU 要件などは扱わず、「どんな軸で分類できるか」「各軸でどんなタイプがあるか」までをまとめる。

> このドキュメントは一般に知られている特性のみを記述する。実測値・断定的な性能評価は含めない（[CLAUDE.md](../../CLAUDE.md) の方針）。
> Phase 1 の現時点では**分類軸のドラフトのみ**で、代表ツールの名前出し（Phase 2）と後続調査タスクの起票（Phase 3）はまだ反映していない。

## 全体像

ボイスチェンジャーは大きく 3 つの軸で位置付けられる。

- **技術アプローチ**: どうやって声を変えるか（信号処理 / ニューラル変換 / ハイブリッド など）
- **提供形態**: どこで動かすか・どう手に入れるか（ローカル OSS / ローカル商用 / クラウド SaaS）
- **モデル形態**: ターゲット声の決め方（話者依存 / 多話者 / Zero-shot / プリセット）

実際のツールはこの 3 軸の組合せで表現できる（例: 「ニューラル変換 × ローカル OSS × 話者依存」）。

```
                   ┌─ 技術アプローチ ─┐
                   │  - 信号処理       │
                   │  - ボコーダ系     │
                   │  - ニューラル変換 │
                   │  - ハイブリッド   │
                   └───────────────────┘
                            │
   ┌─ 提供形態 ─┐           │           ┌─ モデル形態 ──────────┐
   │ - OSS      │ ────────  ◇  ──────── │ - 話者依存            │
   │ - 商用     │                        │ - 多話者 / 非依存     │
   │ - SaaS     │                        │ - Zero-shot           │
   └────────────┘                        │ - プリセット固定      │
                                         └───────────────────────┘
```

## 1. 技術アプローチによる分類

「入力音声をどう加工してターゲットの声に近づけるか」の方式区分。

### 1.1 信号処理ベース (DSP)

- 古典的なピッチシフト・フォルマント変換・EQ などを組み合わせる方式
- 音声を抽象的な特徴量に落とさず、波形/スペクトルを直接操作する
- 一般に低レイテンシで CPU だけでも動くが、別人の声に「化ける」用途には向かないとされる
- 配信用途では「声色のちょい変え」やキャラ加工レイヤーの 1 段として登場することが多い

### 1.2 ボコーダ系 / 古典的声質変換

- ソース音声を励振 + 声道特徴に分解し、フィルタ部分を入れ替えるタイプ
- WORLD などのアナリシス/シンセシスを利用する古典 VC もここに含まれる
- 統計モデル（GMM など）で写像を学習する手法もここに分類する
- ニューラル方式が登場する前の VC 研究の主流系統

### 1.3 ニューラル変換 (Neural Voice Conversion)

入力波形 → 中間表現 → ターゲット話者の波形、をニューラルネットで学習する系統。
内部で使われるアーキテクチャや中間表現で更にサブカテゴリに分かれる。

- **コンテンツ表現 + 話者条件付け系**: 入力から音韻 / コンテンツ表現を取り出し、ターゲット話者の条件を与えて合成する。VITS / so-vits-svc 系などが該当する系統
- **Retrieval ベース系**: 学習したターゲット話者の埋め込みを kNN 等で参照して特徴量を置換する系統
- **拡散モデル (Diffusion) 系**: 拡散ベースのデコーダや声質変換モデルを使う系統
- **End-to-end 波形変換系**: 波形 → 波形を 1 本のネットで学習する系統

### 1.4 ハイブリッド

- 信号処理での前処理（ノイズ抑制 / 音量正規化）と、ニューラル変換を組み合わせる構成
- ニューラル変換の出力にエフェクト（リバーブ / EQ）を後段でかける構成
- 「ボイスチェンジャー製品」と呼ばれるものは、純粋な単一方式ではなく、ほぼハイブリッドで提供される

## 2. 提供形態による分類

「どこで動くか / どう手に入れるか」の区分。配信用途で**そもそも候補に入るか**の足切りに使える軸。

### 2.1 ローカル OSS

- ソースコードが公開されており、自分のマシンで動かす
- GPU を使う前提のものが多い
- モデルの差し替え・自前学習・他ツールとの連携の自由度が高い
- 一方で、セットアップ・依存関係・モデル配布元のライセンスを自分で面倒を見る必要がある

### 2.2 ローカル商用ソフトウェア

- 買い切り / サブスクで配布される、インストール型のデスクトップアプリ
- GUI・サポート・あらかじめ用意されたプリセット声が提供されることが多い
- 内部実装はブラックボックス（信号処理寄り / ニューラル寄り は製品次第）
- ライセンス・配信での利用範囲は製品ごとに規約確認が必要

### 2.3 クラウド SaaS / API

- 変換処理をクラウド側で行い、結果をストリームで返す
- ローカル GPU が不要だが、ネットワーク経由のため遅延・通信品質・利用料が論点
- 配信のリアルタイム入力に乗せられるかは SaaS ごとの提供形態次第（一般には会話/通話向け SDK の有無で判断する）

### 補助カテゴリ: 周辺ツール

声変換そのものではないが、運用上ほぼ必ず組み合わせる層。

- 仮想オーディオデバイス（変換後の音声を配信ソフトに渡すための仮想ケーブル）
- ノイズ抑制 / ゲート（変換前の入力をクリーンにする）
- 配信ソフト・通話ソフト（最終的な配信先）

提供形態の軸ではこれらを「**周辺ツール**」として別立てし、声変換本体の比較とは混同しない。

## 3. モデル形態の軸

「ターゲットの声をどう決めるか」「ターゲットを増やすコスト」の軸。

### 3.1 話者依存型 (Target-specific)

- ターゲット話者ごとにモデル / 重みを学習する
- 1 つの声に対しては品質を高く追い込めるとされる
- 新しい声を増やすたびに学習データの収集と学習が必要
- OSS 系の声質変換は基本この形態が中心

### 3.2 多話者 / 話者非依存型 (Many-to-many)

- 学習済みモデル 1 つで複数のターゲット話者を切り替えられる構成
- 話者埋め込み / 話者 ID を条件として渡す
- 各話者の品質は話者依存型ほど追い込めない場合もあるが、運用は楽になる

### 3.3 Zero-shot / Any-to-any

- リファレンス音声を数秒〜数十秒与えるだけで、その声に変換する
- 個別の追加学習を必要としない（学習済みモデルだけで完結）
- 当然ながら**第三者の声を許諾なく真似る**用途への流用リスクが高く、利用規約・倫理面の判断が重い

### 3.4 プリセット固定型

- ユーザーがモデルを学習・選択するのではなく、ベンダー側があらかじめ用意した「キャラ声」をそのまま使う
- 商用ボイスチェンジャー製品でよく見られる形態
- 中身は話者依存モデル群を切り替えていることが多いが、ユーザーから見ると「プリセットを選ぶだけ」の体験になる

## 4. 配信用途で見るべき主な評価軸

後続の個別調査タスクで揃えるべき観点をリストアップする（本ドキュメントでは値は埋めない）。

| 評価軸 | 何を見るか |
| --- | --- |
| レイテンシ | 入力 → 出力の片道遅延。配信での同期ズレ / 会話の自然さに影響 |
| 音質 | 自然さ・声の安定性・破綻のしにくさ。サンプルレートも明記 |
| 必要 GPU 性能 | VRAM 量 / 推奨世代。GPU 無しで動くか |
| CPU 負荷 | 配信ソフト・ゲームと併走させたときの余裕度 |
| 入出力構成 | 想定する入力デバイス・仮想オーディオデバイス経由の取り回し |
| モデル形態 | §3 のどれに該当するか。新しい声を増やすコスト |
| カスタマイズ性 | 自前モデルの学習可否・読み込み可否・ピッチや声色の調整幅 |
| 配信での利用可否 | 商用配信での利用可否、収益化可否、クレジット要件などライセンス上の制約 |
| エコシステム | 仮想オーディオデバイス・OBS との接続事例、コミュニティ規模、メンテ頻度 |
| コスト | 初期費用 / サブスク / クラウド従量。GPU を持っていない場合の総額 |

## 5. このドキュメントの使い方

- 個別ツール調査ページ（`docs/specs/<tool-name>.md`）の冒頭で、本ドキュメントの分類のどこに該当するかを 1 行で書く
  - 例: 「技術アプローチ: 1.3 ニューラル変換 / 提供形態: 2.1 ローカル OSS / モデル形態: 3.1 話者依存」
- 候補ツールの棚卸し時に、**そもそも候補に入らない方式**（オフライン専用、配信不可ライセンス等）をこの軸で早期に除外する
- 各ツールの「遅延 ◯◯ ms」のような実測値は本ドキュメントには書かず、個別ページ / `experiments/` 配下に記録する
